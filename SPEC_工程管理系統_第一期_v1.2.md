## 1. 背景與目標

Seven 經營多個旅宿工程專案,現行痛點:

- 工程資料(設計圖、現場照片、問題回饋、會議記錄)散落在 LINE 群組與各處,搬移整理耗費大量人力
- 問題回饋依賴口頭與零散訊息,無法追蹤是否銷項,施工後才爆發
- 專案負責人(Seven 或工程夥伴)無法快速掌握「哪些事情還沒完成」

本系統的目標:**將 SOP-01 定義的作業程序自動化**,讓現場人員維持在 LINE 上工作,系統在背後完成結構化、追蹤與提醒。

**成功標準(第一期):** 2026 年 9 月草悟道館與茲心園兩案動工時,所有回饋單/變更單在系統中全程追蹤,LINE 照片與訊息自動掛載至對應工項,Seven 透過儀表板即可掌握兩案的銷項狀態。

## 2. 設計原則(最高優先,衝突時以此為準)

1. **現場零負擔**:設計師、工班、現場同仁只使用 LINE,不安裝、不登入任何新工具。所有結構化工作由系統與專案負責人完成。
2. **系統是 SOP 的自動化**:功能設計必須對應 SOP-01 的某個程序。SOP 沒有的程序,系統不發明。
3. **掛載,而非收集**:每筆資料(訊息/照片/文件)最終都必須掛在結構化錨點上(專案→空間→工項→單據)。無法掛載的資料進入「待確認佇列」,不允許沉入時間流。
4. **多專案原生支援**:上線首日即有兩個以上專案並行,「專案」是資料模型的第一層,不是擴充項。

## 3. 資料模型

### 3.1 實體關係

```
Project(專案/館)
 └─ Space(空間:樓層/房號/公區)
     └─ WorkItem(工項:拆除/水電/防水/泥作/木作/油漆/收尾…)
         └─ Event Stream(事件流,依時間排列):
             ├─ Message(LINE 訊息)
             ├─ Photo(現場照片)
             ├─ FeedbackTicket(圖面問題回饋單)
             ├─ ChangeOrder(工程變更單)
             ├─ Meeting(會議記錄:審圖/交底/檢討)
             └─ Task(待辦事項,含負責人與期限)
```

### 3.2 核心實體欄位定義

**Project**
- id、代碼(館別代碼,見 §5)、名稱、狀態(籌備/審圖/發包/施工/收尾/結案)
- 綁定的 LINE 群組 ID(可多個,見 §3.3)
- 專案負責人(確認者)、目標動工日

**Space**
- id、project_id、區/棟(可空,例:D 區、J 棟)、名稱(例:3F-301、B1 公區)、類型(客房/公區/機房/外部)

**WorkItem**
- id、space_id、工種(enum:拆除/水電/弱電/防水/泥作/木作/鐵工/油漆/設備/收尾/其他,與工班群工種對齊)
- 狀態(未開始/進行中/待複驗/完成)、負責工班(文字即可,第一期不建工班主檔)

**FeedbackTicket(對應 SOP-01 附件 B)**
- 編號(自動生成,規則見 §5)、project_id、space_id、work_item_id(可空,後補)
- 問題描述、照片(多張)、影響等級(A/B/C)、建議做法
- 設計師回覆、回覆期限、狀態(開立→回覆中→已銷項)、逾期旗標
- 完整歷程 log(每次狀態變更、每次推播記錄)

**ChangeOrder(對應 SOP-01 附件 C)**
- 編號、project_id、變更內容、原因(enum:現場條件/設計調整/業主需求/法規)
- 費用影響、工期影響、核准狀態(待核准/已核准/退回)、核准人與時間
- **硬規則:未核准的變更單,狀態不得標為「可施作」**

**Message / Photo**
- 原始內容、LINE 群組來源、發送者、時間戳
- 掛載狀態(未掛載/AI 初判待確認/已確認)
- 掛載目標(project/space/work_item/ticket 之任一或組合)
- AI 初判結果與信心度(供確認介面顯示)

**Meeting**
- 類型(審圖/交底/工地檢討)、日期、參與者、決議事項、待辦事項(展開為 Task)

**Task**
- 內容、負責人、期限、來源(會議/回饋單/手動)、狀態、逾期旗標

### 3.3 LINE 群組與專案的對應

(已確認)每個專案的 LINE 群組結構:**設計師群 + 工班群(依工種分設)+ 內部管理群**。

- 工班群依工種分設,基本工種:**木工、水電、泥作、弱電、鐵工**(依各案實際發包情況開設,不必五群全開;拆除等短期工種可併入相近群或內部群協調)
- 每個群組在後台綁定:project_id + 群組角色(設計師/工班/內部)+ 工種(工班群限定)
- 訊息進入時,依來源群組自動取得**專案與工種兩層**掛載資訊——AI 初判只需補判空間與訊息類型,準確率隨之提升
- 群組角色與工種同時作為 F5 推播的目標依據(催辦設計師 → 設計師群;變更公告 → 受影響工種的工班群,可多選;內部提醒 → 內部管理群)

## 4. 第一期功能規格

### F1:LINE 訊息收集 + AI 初判掛載 + 確認佇列

- 沿用 AM 既有的 LINE OA webhook 收集群組訊息(文字/照片/檔案)。
- 每則訊息進入後,呼叫 Claude API 進行初判,輸出 JSON:
  - 推測的 space(從訊息文字比對房號/樓層關鍵字)
  - 推測的 work_item 工種
  - 訊息類型:進度回報/問題反映/提問/一般對話
  - 若判為「問題反映」→ 建議開立 FeedbackTicket(帶入草稿)
  - 信心度(高/中/低)
- **確認佇列介面**(Web):
  - 分工:**工程夥伴為主要操作者,Seven 負責審核與抽查**;兩人帳號權限相同,但日常處理以夥伴為主
  - 每筆確認記錄操作人與時間;已被處理的項目即從佇列消失,避免重複處理
  - 提供「已確認項目」的審核視圖,供 Seven 抽查夥伴的掛載結果並可修正(修正記錄保留,作為校準依據)
  - 依時間列出待確認訊息,顯示 AI 初判結果
  - 一鍵確認 / 修改掛載目標 / 標為「一般對話」(不掛載,但保留可搜尋)
  - 高信心度項目支援批次確認
- AI 初判的 prompt 中必須帶入該專案的空間清單與工項清單作為上下文。

**驗收標準:** 在測試群組發送 20 則混合訊息(含房號照片、問題反映、閒聊),系統正確收集 100%,AI 初判正確率 ≥ 70%,負責人可在佇列中 3 分鐘內完成全部確認。

### F2:回饋單 / 變更單生命週期管理

- 建立方式:(a) 從確認佇列的「問題反映」一鍵轉開;(b) 負責人手動建立。
- 編號自動生成(§5),不可手動指定、不可重複。
- 狀態機:
  - FeedbackTicket:開立 → 回覆中 → 已銷項(每次轉換記錄時間與操作人)
  - ChangeOrder:待核准 → 已核准/退回
- 逾期提醒:回覆期限前 1 日與逾期當日,系統通知專案負責人(通知負責人即可,是否推播到群組由負責人決定,見 F5)。
- 銷項需附複驗照片或設計師書面回覆,空白銷項不允許。
- 列表視圖:可依專案/狀態/等級/逾期篩選。

**驗收標準:** 完整走一次「開立(含照片)→ 設定期限 → 逾期觸發提醒 → 補回覆 → 附照片銷項」流程,歷程 log 完整無缺漏。

### F3:照片自動歸檔

- 所有經確認掛載的照片,自動以「專案/空間/工項/日期」四層結構歸檔。
- 提供瀏覽介面:選專案 → 選空間 → 依工項與日期瀏覽;支援「此空間全部照片」時間軸視圖。
- 照片保留原圖(工程照片需放大看細節,不得只存壓縮圖)。
- 儲存後端沿用 AM 現有方案;若現有方案不適合存大量原圖,Claude Code 提出替代方案供 Seven 決定(優先考慮 Google Drive,因 Seven 的工作流以 Google Workspace 為主)。

**驗收標準:** 群組傳 10 張不同房號照片,經確認後可在瀏覽介面依空間正確找到全部 10 張,且可下載原圖。

### F4:進度儀表板

- 首頁:各專案卡片,顯示狀態、目標動工日倒數、未銷項回饋單數(A 級標紅)、逾期項目數、待核准變更單數。
- 專案頁:
  - SOP 階段視圖:階段 1/2/3 的完成定義檢核狀態(對應 SOP-01 各階段的「完成定義」條目,由負責人勾核)
  - 未銷項清單(可直接點入單據)
  - 各空間工項狀態矩陣(空間 × 工種的狀態一覽,施工期使用)
- 行動裝置可用(Seven 常在移動中查看)。

**驗收標準:** Seven 打開儀表板 30 秒內能回答:「茲心園現在有幾張 A 級回饋單未銷項?最逾期的是哪一張?」

### F5:雙向推播(系統 → LINE 群組)

- 推播一律是「掛在單據或會議上的動作」,不做自由聊天:
  - **催辦**:對逾期回饋單按「催辦」,系統組裝訊息(單號+問題摘要+照片+期限)推送至指定群組
  - **變更公告**:變更單核准後,一鍵推播至工班群(變更內容+生效日)
  - **會議記錄發布**:會議記錄完成後推播摘要+待辦清單
- 每次推播記入對應單據的歷程 log(時間、內容、目標群組)。
- **額度控管**:LINE OA 推播有月額度,系統顯示本月已用量;推播僅限上述三種場景,不做自動群發。
- 推播前顯示預覽,負責人確認後才發送。

**驗收標準:** 對一張逾期單執行催辦,目標群組正確收到格式完整的訊息,單據歷程中出現該次推播記錄。

## 6. 與現有 AM 架構的銜接

- LINE OA webhook、群組訊息收集:**沿用現有實作**,新增「依群組 ID 對應專案」的路由層。
- HOZO AM 與 Seven AM 的既有功能**不得受影響**;工程模組以獨立模組/命名空間開發,共用基礎設施(webhook 接收、儲存、部署)。
- AI 初判使用 Claude API(模型與呼叫方式由 Claude Code 依現有架構決定,成本敏感場景可用 Haiku 級模型)。
- **Claude Code 的第一項任務**:盤點現有架構(儲存方案、部署環境、webhook 流程),輸出「銜接方案說明」供 Seven 確認,再進入 M1 開發。
