# AM 任務下一步判斷規則

本文件定義「下一步」由誰判斷、何時判斷、寫到哪裡，以及 User UI 的責任邊界。

## 1. 核心原則

`下一步` 是任務控制資料，不是 User UI 呈現邏輯。

User UI 不應在渲染任務頁或任務看板時自行推論下一步。User UI 只應讀取 Notion 任務資料庫或任務內文中已寫入的下一步，然後一致呈現。

## 2. 負責判斷下一步的單位

下一步應由下列流程判斷並寫回 Notion：

- 每小時 LINE task reconciliation。
- 新任務建立流程。
- 既有任務更新流程。
- 會議 checkbox / action item 同步流程。
- 報告候選項目與報告決策寫回流程。
- 使用者在 User UI 或 Notion 中明確手動編輯任務。
- 任務校準或判斷規則更新後的重判流程。

## 3. 判斷時機

應在下列情境更新下一步：

- 新訊息形成新任務時。
- 新訊息補充、回答、完成、阻塞或改變既有任務時。
- 會議記錄產生任務或更新任務時。
- 報告中使用者確認候選項目或決策寫回時。
- 任務狀態、負責人、期限、專案目標或完成條件改變時。
- 使用者明確修改下一步時。

## 4. 寫入位置

下一步應寫入專案本地 Notion 任務資料庫。

資料庫來源：

| 專案 | 任務資料庫 | 環境變數 |
| --- | --- | --- |
| HOZO AM | HOZO AM 總控任務庫 | `HOZO_TASKS_DATA_SOURCE_ID` |
| SevenAM | SevenAM / 7AM 總控任務庫 | `SEVEN_TASKS_DATA_SOURCE_ID` |
| 共用模板 | 專案本地總控任務庫 | `PROJECT_TASKS_DATA_SOURCE_ID` / `PROJECT_*` 對應欄位 |

優先順序：

1. `下一步` 欄位。這是 User UI 的標準讀取欄位。
2. `下一步給負責人` 欄位。這是舊版/LINE 查待辦相容欄位；若仍存在，應逐步同步回 `下一步`。
3. 任務內文中的 `下一步` 區塊，例如 `## 三、下一步`，只作為過渡期備援。
4. 任務更新紀錄或狀態變更紀錄中的下一步摘要，只作為最後備援。

User UI 讀取時應依同樣優先順序呈現。

長期標準：

- 新 schema 應使用 `下一步`。
- 既有流程若仍寫入 `下一步給負責人`，應同步或遷移到 `下一步`。
- User UI 不應直接從來源對話推論下一步，只能讀取上述欄位或任務內文中已存在的資料。

## 5. User UI 責任

User UI 應該：

- 顯示 Notion 中已存在的下一步。
- 在沒有下一步時顯示 `未設定` 或 `待判斷流程補齊`。
- 可在專案支援時提供手動編輯欄位，並把使用者輸入寫回 Notion。
- 顯示最後更新時間或編輯備註，若資料可用。

User UI 不應該：

- 根據任務內容自行產生下一步。
- 根據來源對話自行改寫下一步。
- 因為欄位空白就推論下一步。
- 把 UI placeholder 當成任務資料寫回。

## 6. 檢驗要求

User UI 全頁檢驗應檢查：

- 任務頁是否只呈現 Notion 已有的下一步。
- 沒有下一步的任務是否顯示空狀態。
- 產生器是否沒有以來源文字即時計算下一步。
- 可編輯下一步時，是否明確寫回 Notion 並留下編輯備註。
