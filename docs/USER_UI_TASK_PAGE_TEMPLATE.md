# AM User UI 任務內頁模板

這是 AM 類專案共用的任務詳情頁模板標準。

任務頁應由專案本地任務資料、LINE 訊息、會議記錄、附件與報告資料產生，不得逐頁手動修改。

## 資料欄位

每一個任務頁在資料可用時，應由下列欄位渲染：

| 欄位 | 意義 |
| --- | --- |
| `projectName` | 專案顯示名稱，例如 HOZO AM 或 SevenAM。 |
| `taskTitle` | 人類可讀的任務標題。 |
| `taskProject` | 此任務支援的專案目標或專案卷宗。 |
| `status` | 目前任務狀態。 |
| `owner` | 目前負責人或負責角色。 |
| `priority` | 優先級或風險標籤。 |
| `deadline` | `截止日`。 |
| `deadlineBasis` | `期限依據`。 |
| `nextFollowUpDate` | `下次追蹤日`。 |
| `overdueStatus` | `逾期狀態`。 |
| `nextStep` | 從專案本地總控任務庫讀取的下一步內容。標準欄位為 `下一步`，舊版相容欄位為 `下一步給負責人`。User UI 不自行判斷下一步。 |
| `judgmentSummary` | 從 Notion 任務欄位或任務內文讀取的 AM 判斷摘要。User UI 不重新推論。 |
| `sourceType` | `LINE`、`meeting`、`report`、`manual` 或 `unknown`。 |
| `sourceSummary` | 人類可讀的來源摘要。 |
| `sourceUrl` | 專案本地 User UI 來源頁或 Notion 紀錄。 |
| `evidenceMessages` | 已匹配的 LINE 訊息或報告證據。 |
| `meetingEvidence` | 已匹配的會議記錄內容。 |
| `mediaItems` | 圖片、照片、檔案、PDF、影片、語音或附件連結。 |
| `auditNotes` | 來源標記、同步識別碼、寫回備註與更新歷史。 |

## 必要區塊順序

### 1. 任務標頭

顯示：

- 任務標題。
- 專案名稱。
- 狀態。
- 負責人。
- 專案/目標。
- 截止日與下次追蹤日。
- 回到所有任務的連結。

規則：

- 標題必須人類可讀。
- 標題不得包含 Notion id、LINE id、sync id、message id 或其他技術識別碼。
- 畫面上的專案名稱必須符合正在產生頁面的子專案。

### 2. 任務控制區

在專案支援時，顯示可編輯或可審閱的任務控制欄位：

- 狀態。
- 負責人。
- 專案/目標。
- 下一步。
- 截止日。
- 期限依據。
- 下次追蹤日。
- 逾期狀態。
- 判斷摘要。

規則：

- 每個任務頁都應顯示同一組欄位。
- 沒有資料時應顯示清楚的空狀態，不得無聲消失。
- 狀態為 `封存`、`已封存` 或 `Archived` 的任務不得由 User UI 產生器帶入所有任務清單、專案任務清單或任務頁輸出。封存任務保留在專案本地 Notion 任務庫，不作為日常 User UI 工作清單項目。

### 3. 來源摘要

顯示建立或更新任務的上游來源。

LINE 衍生任務：

- 標籤：`資料來源：LINE 對話`。
- 來源對話群組。
- 專案本地 User UI 對話頁連結。
- 可用時顯示相關時間範圍。
- 可用時顯示相關發話者。

會議衍生任務：

- 標籤：`資料來源：會議記錄`。
- 會議標題。
- 會議日期。
- 會議頁連結。
- checkbox/action item 文字。
- source marker 與 sync id。

規則：

- 會議衍生任務不得 fallback 成 `來源對話群組：LINE 對話群組`。
- 原始 Notion 訊息 URL 應轉換成人類可讀的來源摘要。

### 4. 證據時間線

用一致格式顯示來源證據。

LINE 證據：

- 必須回查專案本地 `LINE 對話主檔` 或 `LINE 對話組檔`。
- 必須沿用 LINE 對話主檔內文樣式，不得使用任務頁自訂對話卡片取代。
- 每則訊息標題列包含時間、群組、發話者與訊息類型。
- 訊息內容直接接在標題列下方，並保留原始換行。
- 圖片訊息必須在標題列下方直接顯示圖片。
- 可用時連到來源訊息。
- 連到對話頁。

LINE 對話引用的完整呈現標準定義於：

```text
D:\Codex_project\AM_Core\docs\LINE_CONVERSATION_RENDERING_STANDARD.md
```

會議證據：

- 會議標題與日期。
- 會議摘要或摘錄。
- 決議或 checkbox 項目。
- 來源頁連結。

規則：

- 不得顯示舊版獨立 `來源證據與對話記錄` 區塊。
- 不得顯示原始 `判斷補充文字` fallback 卡片。
- LINE 對話引用不得混用其他舊版對話卡片、泡泡或時間線樣式。
- 必須保留足以稽核任務萃取與狀態變更的來源脈絡。

### 5. 媒體與附件

顯示與證據相關的所有媒體與檔案。

規則：

- 圖片/照片證據顯示為可點擊縮圖。
- LINE 圖片訊息在引用對話內容時，必須依 LINE 對話主檔樣式直接顯示在該則訊息標題列下方。
- 檔案/PDF/影片/語音顯示為連結。
- 媒體可以從下列來源匹配：
  - LINE message media。
  - 專案本地附件紀錄。
  - 封存文字中的 LINE message id，例如 `圖片：617...` 或 `[image] 617...`。
- 如果媒體不存在，顯示包含 message id 與來源資訊的 placeholder。

### 6. 目前判斷

用使用者可理解的語言顯示 AM 的判斷：

- 為什麼這是一個任務。
- 它支援哪個專案目標。
- 負責人或負責角色是誰。
- 哪些事情已完成。
- 哪些事情仍未解決。
- 目前狀態由哪些證據支持。

規則：

- 此區塊應解釋任務，不應暴露內部 prompt 或機器語言。
- 如果信心不足，應明確說明需要確認的內容。

### 7. 下一步

顯示 Notion 任務資料庫中已寫入的下一步欄位或任務內文中的下一步段落。

規則：

- User UI 不負責判斷、生成或改寫下一步。
- 下一步的判斷應在任務建立、每小時 LINE 檢查、任務狀態更新、會議同步或報告寫回時完成，並寫入 Notion 任務資料庫。
- User UI 只負責讀取並呈現 Notion 中的下一步欄位。標準來源是專案本地總控任務庫的 `下一步` 欄位；若專案仍有舊欄位 `下一步給負責人`，可作為相容備援。
- 如果 Notion 沒有下一步，User UI 顯示 `未設定` 或 `待判斷流程補齊`，不得自行推論。
- 如果使用者在 User UI 修改下一步，該修改是明確資料編輯，應寫回 Notion 並留下編輯備註；不是由 User UI 自動判斷。

### 8. 稽核與寫回紀錄

顯示：

- source marker。
- sync id。
- 更新歷史。
- 狀態變更證據。
- 可用時顯示寫回結果。

規則：

- 技術識別碼可以出現在此區供稽核，但不得出現在任務標題。
- 稽核資訊不得壓過主要任務內容。

## 空狀態規則

欄位沒有資料時：

- 依情境使用 `未設定`、`未找到來源` 或 `目前沒有附件`。
- 如果隱藏區塊會造成頁面不一致，就不得隱藏。
- 不得捏造不存在的資料。

## 跨專案規則

- HOZO AM 產生頁面必須顯示 HOZO AM，不得顯示 SevenAM。
- SevenAM 產生頁面必須一致顯示 SevenAM 或 7AM，不得顯示 HOZO AM。
- 任一專案的資料不得出現在另一個專案。
- 生成頁面必須從各專案自己的 `.env` 與 Notion data sources 重新產生。

## 驗證清單

每次完整任務頁產生都必須驗證：

- 每個任務頁都使用相同必要區塊順序。
- 任務標題不包含技術識別碼。
- 沒有舊版 `來源證據與對話記錄` 區塊。
- 沒有原始 `判斷補充文字` fallback 卡片。
- LINE 衍生任務在可用時連到對話頁。
- 會議衍生任務顯示會議證據，且不顯示 LINE fallback。
- 封存文字中的 image id 在媒體存在時會渲染縮圖。
- 所有本地圖片/檔案連結都指向存在的生成媒體檔。
- 提到圖片/照片但沒有媒體的頁面會列入例外清單。
- 每個 active task 都顯示期限控制。
- 畫面上的專案名稱符合正在產生的子專案。
