# AM User UI 規則 - 2026-06-09 整合版

本文件彙整 2026-06-09 討論與實作過的所有 User UI 要求。

這份文件是下一階段實作的規則來源：

1. 更新共用模板與產生器。
2. 分別套用到 HOZO AM 與 SevenAM。
3. 重新產生每個專案的全部 User UI 頁面。
4. 產出全頁面檢驗報告。

核心目標是避免「看到哪一頁有問題就修哪一頁」。User UI 的行為必須來自共用規則、共用模板、產生器邏輯與全專案檢驗，而不是個別頁面手動修補。

## 1. 運作模式

### 1.1 目前 User UI 的產生方式

目前 User UI 是在各專案資料夾中產生靜態 HTML 頁面。

例如：

- `docs/user-ui-connected-preview.html`
- `docs/user-ui-task-0.html`
- `docs/user-ui-line-0.html`

這些頁面不是圖片檔，但它們是「產出結果」。當規則文件改變時，這些頁面不會自動更新。只要共用模板或產生器改變，就必須重新產生該專案的所有 User UI 頁面。

### 1.2 正確工作方式

User UI 的改善不得透過直接修改某一個已產生的任務頁完成。

正確流程：

1. 在 AMCore 寫入或更新規則。
2. 更新共用模板或產生器。
3. 分別套用產生器到每一個子專案。
4. 重新產生該子專案的所有頁面。
5. 執行全頁面檢驗。
6. 依頁面類型與例外清單回報結果。

已產生的 HTML 頁面應視為可丟棄的輸出。真正持久的來源是：

- AMCore User UI 規則。
- AMCore 共用模板與產生器。
- 各專案自己的 Notion 資料。
- 各專案重新產生的 HTML。
- 各專案自己的檢驗報告。

## 2. 頁面清單與最終要求

### 2.1 專案總覽頁

用途：

- 提供使用者快速進入專案管控的入口。
- 連到專案、任務、LINE 對話、會議、報告、判斷規則、環境資料、升級版本與開發設定。

最終要求：

- 只顯示該專案自己的資料。
- 不得混入 HOZO AM 與 SevenAM 彼此的資料。
- 畫面文字必須使用正確專案名稱。
- HOZO AM 頁面不得殘留 `SevenAM`；SevenAM 頁面不得殘留 `HOZO AM`。
- 密碼、Token、Secret 與憑證必須遮蔽。

### 2.2 所有專案頁

用途：

- 顯示專案卷宗，以及支撐每個專案目標的任務。

最終要求：

- 專案列必須連到該專案自己的專案紀錄。
- 專案底下的任務清單，必須與「所有任務頁」使用同一套任務標題與狀態格式。
- 專案證據、附件與會議引用都必須留在該專案內，不得跨專案混用。

### 2.3 所有任務頁

用途：

- 提供容易掃描的總控任務清單。

最終要求：

- 任務標題必須是人看得懂的名稱。
- 任務標題不得包含 Notion page id、LINE id、user id、room id、message id、sync id 或其他技術識別碼。
- 預設任務表格視圖應隱藏 `來源原文`。
- `來源原文` 欄位可以留在資料庫中作為稽核備用，但不應顯示為預設表格欄位。
- 狀態為 `封存`、`已封存` 或 `Archived` 的任務不得帶入所有任務清單，也不得出現在專案下的任務清單或任務頁輸出中。封存任務只保留在專案本地 Notion 任務庫作為歷史紀錄。
- 每一列任務至少應顯示可行動所需欄位：專案、狀態、負責人、下一步、截止日、下次追蹤日、逾期狀態，以及任務內頁連結。
- `下一步` 欄位必須直接來自專案本地 Notion 總控任務庫。標準欄位為 `下一步`；舊版相容欄位為 `下一步給負責人`；最後才可讀任務內文中的已寫入下一步段落。User UI 不自行判斷下一步。

相關規則版本：

- `AM-IMP-2026.0608.15`
- `AM-IMP-2026.0608.19`
- `AM-IMP-2026.0609.05`
- `AM-IMP-2026.0610.04`

### 2.4 個別任務內頁

用途：

- 每一個任務頁都是任務卷宗。
- 任務頁應呈現這是什麼任務、為什麼存在、來源是什麼、狀態如何改變、有什麼圖片或檔案支持，以及 Notion 中已記錄的下一步。

最終要求：

- 每一個任務頁都必須使用同一個共用模板。
- 每一個任務頁都必須有相同的區塊順序與標籤。
- 任務正文必須顯示真正的上游來源：LINE 對話或會議記錄。
- 對話群組與關聯頁面標籤，在可用時應連到該專案自己的 User UI 對話頁。
- 任務頁凡引用 LINE 對話內容，必須回查專案本地 `LINE 對話主檔` 或 `LINE 對話組檔`，並使用該主檔內文樣式呈現，不得改用其他對話卡片、泡泡或時間線樣式。
- 會議衍生任務必須連到會議證據，不得使用泛用 LINE fallback。
- 舊版獨立 `來源證據與對話記錄` 區塊不得出現。
- 舊版原始 `判斷補充文字` fallback 卡片不得出現。
- 舊的來源訊息 URL 應轉換為可讀的來源摘要。
- 頁面必須用一致順序顯示：狀態、負責人、專案、期限控制、下一步、來源摘要、證據、圖片/附件、稽核資訊。
- User UI 不負責產生任務內容、狀態判斷、期限判斷或下一步判斷；這些決策應由任務判斷流程寫入 Notion 後，User UI 再讀取呈現。

任務頁的正式區塊順序定義於：

```text
docs/USER_UI_TASK_PAGE_TEMPLATE.md
```

相關規則版本：

- `AM-IMP-2026.0609.03`
- `AM-IMP-2026.0609.05`
- `AM-IMP-2026.0609.09`
- `AM-IMP-2026.0609.10`

### 2.5 任務頁來源證據

用途：

- 保留 AM 建立或更新任務的理由與來源。

最終要求：

- LINE 證據必須使用 `LINE 對話主檔` 或 `LINE 對話組檔` 的內文格式：
  - 第一行為彩色標題列，包含日期時間、群組名稱、發話者名稱與訊息類型。
  - 下一行直接顯示實際訊息內容。
  - 圖片訊息必須在標題列下方直接顯示圖片。
- 一般 LINE 來源訊息標題列應沿用主檔藍色樣式；Seven Jr.、HOZO Jr. 或系統助理訊息應沿用主檔橘色樣式。
- LINE 證據應可讀，不得把原始 Notion id 當作主要顯示標籤。
- 如果 LINE 訊息以 `圖片：617...` 或 `[image] 617...` 這類 message id 表示，任務頁必須在專案本地訊息/媒體紀錄存在時，把它解析回圖片或媒體。
- 如果證據無法取得，頁面必須顯示可稽核的 placeholder，不得無聲刪除來源。
- LINE 對話引用的完整樣式標準定義於 `docs/LINE_CONVERSATION_RENDERING_STANDARD.md`。

### 2.6 任務頁圖片與附件

用途：

- 讓使用者能直接檢視造成或支持任務的圖片、照片與檔案。

最終要求：

- 不得把 LINE 圖片、照片或檔案證據簡化成純文字。
- 圖片與照片證據在有本地預覽檔或外部圖片 URL 時，必須呈現為可點擊縮圖。
- 若圖片來自 LINE 對話引用，圖片必須直接出現在該則訊息標題列下方，與 LINE 對話主檔內文樣式一致。
- 非圖片附件必須呈現為檔案連結或附件紀錄連結。
- 媒體來源可以是：
  - LINE 訊息紀錄上的 media。
  - 專案本地附件資料庫。
  - 封存文字中的 LINE message id。
- AMCore 不得把任何專案的 live media 複製到共用 repository。
- 如果圖片或檔案缺失，仍須顯示 media placeholder、message id、群組、發話者、時間，以及可用的附件紀錄連結。

2026-06-09 HOZO AM 已知檢驗結果：

- `弱電與資訊維修費用佐證確認（UOF 表單與 5 月急件）` 這個任務正文中有兩個 LINE 圖片 id，但原本沒有縮圖。
- 已更新產生器，讓封存文字中的圖片 id 可以解析為縮圖。
- 重新產生後，該任務頁已顯示：
  - `user-ui-media/617507123290964004.jpg`
  - `user-ui-media/617507397195268573.jpg`
- 另外三個 HOZO 任務頁只是概念上提到照片，但沒有具體圖片 message id 或附件紀錄：
  - `交接素材整理：整併 Google 雲端硬碟、房型照片、591 刊登素材。`
  - `自動化點交與電子合約：整併 LINE 回傳照片、證件/房況存檔、電子合約 PDF 與公司大小章。`
  - `通路上架、月租規範與帶看內容`

這三個是資料來源例外，不是模板例外。要附上圖片前，必須先有實際媒體來源。

### 2.7 會議記錄頁

用途：

- 會議記錄是第一級任務來源與知識來源。

最終要求：

- 會議頁應顯示會議名稱、日期、摘要、決議、checkbox/action items，以及可用的關聯任務。
- 會議紀錄中的每一個 checkbox 都是已確認的任務來源。
- 會議討論與決議應保留作為執行任務時的背景知識。

相關規則：

- 會議 checkbox 項目即為任務。
- 會議記錄是來源證據，不是一般備註。

### 2.8 會議衍生任務頁

用途：

- 避免會議任務被錯誤標示為 LINE 任務。

最終要求：

- 如果任務來自會議 checkbox、action item，或 `meeting:<meetingPageId>:<itemId>` 這類 sync id，任務頁必須將來源分類為會議證據。
- 來源區塊顯示 `資料來源：會議記錄`。
- 關聯頁面在可用時連到會議記錄。
- 任務頁在可用時顯示會議名稱、日期與有用的會議內文摘要。
- checkbox/action item 必須保留為 `行動項目`。
- source marker 與 sync id 必須保留供稽核。
- 會議衍生任務不得顯示 `來源對話群組：LINE 對話群組`。

相關規則版本：

- `AM-IMP-2026.0609.10`

### 2.9 LINE 群組與訊息頁

用途：

- 讓使用者可以回查任務背後的原始對話歷史。

最終要求：

- LINE 群組頁必須顯示群組名稱、專案關聯、近期訊息、訊息類型、發話者、時間、任務狀態與媒體。
- LINE 群組頁與任何引用 LINE 對話的區塊，必須使用 `docs/LINE_CONVERSATION_RENDERING_STANDARD.md` 的同一套 renderer。
- 圖片訊息必須呈現為可點擊縮圖。
- 檔案訊息必須呈現為檔案連結。
- 任務證據卡必須重用同一套媒體 renderer。
- 對話頁可以顯示「目前沒有附件」，但不得隱藏已經有本地預覽檔的圖片訊息。

### 2.10 附件與檔案頁

用途：

- 提供一個地方檢視與管理專案本地檔案、PDF、圖片、OCR、解析狀態與附件狀態。

最終要求：

- 附件紀錄必須留在專案本地。
- 使用者可以在可用時開啟附件，並看到關聯專案或關聯對話。
- 附件動作可依專案 runtime 支援 save、convert、archive 或修改狀態。
- OCR/PDF/image 解析狀態在可用時應顯示。
- 任一專案不得顯示另一個專案的附件資料。

### 2.11 報告入口與報告紀錄頁

用途：

- User UI 只負責讓使用者查看、開啟或回查報告紀錄。
- User UI 不負責定義各時段報告如何產生、候選項目如何判斷、或報告中的決策動作如何寫回。

最終要求：

- 報告入口可顯示報告名稱、時段、產生時間、狀態、開啟連結與寫回狀態。
- 報告紀錄必須維持專案本地資料邊界。
- 報告頁若由 User UI 連入，畫面上的專案名稱必須正確。
- User UI 規則不得硬編 08:00、10:00、13:00、17:00、20:30 報告的產生邏輯。
- 各時段報告的內容順序、候選任務判斷、確認動作與寫回規則，應維護在報告規則文件：

```text
D:\Codex_project\AM_Core\docs\REPORT_SLOT_RULES.md
```

相關規則版本：

- 報告規則文件會引用 `AM-IMP-2026.0608.11` 與 `AM-IMP-2026.0609.04`。

### 2.12 任務判斷規則頁

用途：

- 讓任務萃取與任務狀態判斷邏輯可見、可審閱、可補充。

最終要求：

- 頁面標籤為 `任務判斷規則`。
- 頁面顯示：
  - AMCore 共用規則。
  - 專案專屬規則。
  - 只屬於目前專案的學習校準規則。
  - 使用者從 User UI 手動新增的規則。
- 歷史校準案例可留在 Notion 供稽核，但有用的規則內容應呈現在規則頁，不應另外成為一個主要導航頁。
- 手動新增規則會送到該專案自己的 control API，並寫入該專案自己的 judgment rules database。
- 手動新增規則不只是 User UI 顯示資料；每小時 LINE 任務判斷與其他任務判斷流程必須依 `docs/TASK_JUDGMENT_RULE_LOADING_STANDARD.md` 載入最新規則。
- HOZO AM 規則不得寫入 SevenAM；SevenAM 規則不得寫入 HOZO AM。

相關規則版本：

- `AM-IMP-2026.0609.06`
- `AM-IMP-2026.0609.07`

### 2.13 Environment data 頁

用途：

- 以可讀且安全的方式顯示專案 runtime 設定。

最終要求：

- Environment rows 依下列群組排序：
  1. Credential
  2. Secret
  3. Notion ID
  4. Database ID
  5. LINE
  6. Report
  7. URL
  8. Runtime
  9. Config
- `USER_UI_USERNAME` 必須顯示在 `USER_UI_PASSWORD` 正上方。
- 敏感值必須遮蔽。
- 未來若有 reveal action，必須要求 Admin 權限並留下 audit log。
- AMCore 不得儲存 live project `.env` values。

相關規則版本：

- `AM-IMP-2026.0609.08`

### 2.14 升級版本頁

用途：

- 讓使用者知道每個 AMCore 改善版本在各專案的安裝狀態。

最終要求：

- 顯示版本 id、改善名稱、狀態、日期、參考紀錄與驗證結果。
- 只允許使用狀態：Proposed、Ready、Installed、Deployed、Blocked、Deprecated。
- 如果某版本在一個專案是 active，另一個專案也應有對應紀錄；除非該版本明確排除或被標示為 Blocked。
- 不得在未完成該專案 production Render service 驗證前標示為 Deployed。

### 2.15 開發者與管理設定

用途：

- 提供專案本地控制功能，同時避免一般使用者看到秘密資訊。

最終要求：

- 專案使用者只能看到該專案自己的資料。
- Admin 使用者在系統支援時可以看跨專案設定。
- Secrets 預設必須遮蔽。
- 高風險寫入動作必須有明確控制與 audit record。

## 3. 模板要求

每一個共用模板都必須分離：

- 呈現規則。
- 資料欄位。
- 資料來源匹配規則。
- 驗證規則。

AMCore 的模板不得硬編任何專案資料、客戶訊息、秘密、production database id 或 live attachment records。

子專案可以產生靜態 HTML，但這些生成檔必須視為輸出結果。模板改變後必須重新產生。

## 4. 驗證要求

每次完整 User UI 產生都必須產出報告，至少包含：

- 總產生頁數。
- 已檢查任務頁數。
- 已檢查 LINE 頁數。
- 已檢查會議頁數。
- 已檢查報告入口與報告紀錄連結數。
- 含媒體證據的頁數。
- 提到圖片/照片但缺少媒體的頁面。
- 缺失的本地媒體檔案。
- 使用舊版區塊的任務頁。
- 專案名稱錯誤的任務頁。
- 會議衍生任務被誤標為 LINE 任務的頁面。
- 缺少期限控制的任務頁。
- Environment rows 中未遮蔽的敏感值。
- 跨專案資料混用警告。
- 通過/失敗摘要與例外清單。

使用者應審閱例外清單，而不是逐頁檢查所有頁面。
