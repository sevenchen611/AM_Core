總控任務判斷與專案歸屬流程圖

這份流程圖描述任何 LINE 對話或訊息進入 AM 系統後，如何從原始訊息一路判斷成 「不是任務」、「待確認訊號」、「總控任務」、「專案進度」、「風險/決策」， 並標出控制者可以在哪些時機介入，指定專案、專案目標或任務目標。

自動接收/保存 系統判斷 控制者可介入 寫入正式資料庫 高風險需確認
目前建議原則： 控制者可以在 LINE 指令、群組/權責設定、每日與各時段回報、任務庫人工修正這幾個位置介入。 每天的回報報表應該成為固定介入窗口，讓控制者把訊息重新指定到正確專案、專案目標或任務目標。

主流程圖

LINE / Notion 收件層

收到 LINE event

文字、附件、貼圖、join/leave、postback 或其他事件。

找/建對話主檔

依 User ID、Group ID、Room ID 建立或更新對話。

建立訊息紀錄

保存文字、來源、發話者、原始 payload、時間。

分流附加資料

附件進附件表；助理指令進 Codex command queue。

更新群組治理資料

群組選項、群組成員選項、權責候選應同步刷新。

進入判斷佇列

等待 hourly sync 或報表生成時被判斷。

系統判斷層

訊息是否有行動價值？

排除寒暄、純通知、重複訊息、低訊號內容。

是總控訊號嗎？

任務、進度、風險、決策、等待回覆、責任歸屬

分類訊號類型

任務 / 進度 / 風險 / 決策 / 會議行動 / 追蹤提醒。

推論專案歸屬

依明確指令、群組權責、關鍵字、會議專案、歷史脈絡排序判斷。

推論目標與下一步

整理任務名稱、完成目標、負責人、期限、下一步。

信心足夠嗎？

不足則進入待確認，而不是直接寫成正式任務。

控制者介入層

LINE 即時指令

例如：這件歸到「財務」，目標是「完成付款確認」。

群組/權責設定

指定某群組屬於哪個總控專案、主要負責人是誰。

候選任務確認

在任務候選、跟催清單或回報報表中確認是否建立任務。

專案與目標修正

把任務改歸到特定專案、專案目標、任務目標或完成定義。

每日/時段報表介入

08:00、10:00、13:00、17:00、20:30 都應可確認、改派、補目標。

人工覆寫成準則

控制者修正後，應回寫為群組、專案或關鍵字規則。

正式寫入層

非任務留存

只保留在訊息紀錄與對話頁，作為脈絡。

待確認訊號

進報表、候選清單或指令佇列，等待控制者確認。

建立總控任務

寫入總控任務庫，帶入專案、狀態、優先級、來源頁。

建立進度/風險/決策

必要時同步專案進度報表庫、風險與決策庫。

回報與追蹤

下一次報表依任務狀態、目標確認、責任人更新追蹤。

完成或封存

完成後標記、摘要、保留來源鏈結與決策紀錄。

如何判斷是否為「總控任務」

會成為任務的訊號

  • 有明確要做的動作：處理、確認、追蹤、整理、回覆、付款、簽核。
  • 有責任人、對象、期限、等待回覆或下一步。
  • 影響專案進度、客戶承諾、財務、合約、交付或營運風險。
  • 控制者明確說「這是任務」、「幫我列入總控」、「下次報告追蹤」。

不應直接成為任務

  • 單純寒暄、收到、貼圖、沒有行動指向的資訊。
  • 重複訊息或已經有同名任務正在追蹤。
  • 脈絡不足，無法判斷專案或下一步。
  • 高風險內容需要確認，不能直接執行或承諾。

介於中間的訊號

  • 看起來重要但缺少目標：進入候選清單。
  • 看起來像決策但沒有結論：進報表請控制者判斷。
  • 看起來像進度但缺少完成標準：請負責人口述目標。
  • 同一事件有多則訊息：先合併成事件結論再建立任務。

專案、專案目標、任務目標的歸屬順序

優先序 來源 說明 控制者是否可介入
1 控制者明確指示 例如「這件歸到公司治理」、「目標是完成公司設立文件」、「負責人是 Maggie」。 可以，最高優先。
2 權責定義與群組設定 群組已綁定總控專案、主要對話群組、主要負責人時，用這些設定作為預設。 可以，在 `權責定義`、`LINE 群組選項` 中調整。
3 既有任務或會議脈絡 訊息若引用某任務、會議或既有專案，沿用其專案與目標。 可以，在任務庫或會議紀錄修正。
4 關鍵字與內容判斷 依財務、合約、工程、品牌、營運、系統等關鍵字推論。 可以，透過報表修正後回寫成規則。
5 未分類 / 待確認 信心不足時不直接放錯專案，而是標為待確認。 可以，應在下一次報表或候選清單中處理。

控制者可以介入的時機

介入點 可以做什麼 應寫回哪裡
LINE 當下指令 直接說明任務歸屬、專案、目標、負責人、期限。 Codex 指令佇列、總控任務庫、權責規則。
群組第一次建立後 指定群組屬於哪個總控專案，是否納入監控。 LINE 群組選項、權責定義。
08:30 早報 確認今日目標、昨日未完成事項是否轉成任務。 總控任務庫、目標欄位。
10:00 / 13:00 / 17:00 跟催報表 確認候選任務、改派專案、補任務目標、要求負責人回覆。 總控任務庫、權責定義、通知候選。
20:30 每日報告 把當天事件結論定稿，確認哪些要追蹤、哪些不追蹤。 總控任務庫、專案進度報表庫、風險與決策庫。
任務庫人工修正 改專案、狀態、優先級、負責人、完成目標。 總控任務庫；必要時同步回報表與規則。

報表是否都可以介入？

設計上應該可以。五個固定報表時段都應該是控制者介入窗口： 08:30 先定今日目標，10:00 / 13:00 / 17:00 做候選任務與跟催確認，20:30 做事件結論與隔日追蹤定稿。

目前程式已經有報表 preview、send、approve 的控制 API 基礎； AMCore 已將報表介入動作定義成 `AM-IMP-2026.0608.11` 標準。 每個候選項都應提供： `建立任務`、`不是任務`、`改專案`、`指定專案目標`、`指定任務目標`、`要求負責人口述目標`。
