# AM 各時段報告規則

本文件定義 AM 各時段報告的產生邏輯、呈現順序、候選項目處理方式與寫回規則。

這份文件屬於 Reporting / Report Governance，不屬於 User UI 規則。User UI 可以提供報告入口、報告紀錄連結與報告狀態顯示，但不得在 User UI 規則中定義各時段報告的產生邏輯。

## 1. 規則邊界

### Report 規則負責

- 各時段報告的產生目的。
- 每個報告時段要顯示哪些區塊。
- 候選訊息、候選任務、風險、目標與待確認事項如何出現在報告中。
- 使用者可以對報告候選項目採取哪些動作。
- 報告決策如何寫回任務、專案、進度報告或判斷規則。
- 報告送出、重試、紀錄與驗證。

### User UI 規則只負責

- 顯示報告入口或報告紀錄列表。
- 連到專案本地報告頁或報告紀錄。
- 顯示報告是否已產生、是否已送出、是否已寫回。
- 確保畫面上的專案名稱與資料邊界正確。

## 2. 共同報告原則

- 每個報告都是任務管控的一個介入點。
- 報告不只是摘要，必須能協助使用者決定：建立任務、更新任務、確認不是任務、改專案、補目標、或要求負責人說明目標。
- 報告中的候選項目必須可以被決策，不應只顯示無法操作的說明文字。
- 無法被使用者確認、改變或寫回的純說明規則，應放在文件或 AMCore 規則中，不應放在報告 UI。
- 報告資料與寫回結果都必須保留在該專案本地，不得跨 HOZO AM / SevenAM 混用。

## 3. 標準報告介入動作

報告候選項目應使用標準 action key。顯示文字可以是中文，但寫回資料應使用穩定 key。

標準動作來源：

```text
D:\Codex_project\AM_Core\config\report-intervention-actions.json
```

主要動作：

| 顯示意圖 | 寫回意義 |
| --- | --- |
| 建立任務 | 將候選項目建立為任務。 |
| 不是任務 | 判定候選項目不需要進入任務庫。 |
| 改專案 | 重新指定候選項目所屬專案。 |
| 指定專案目標 | 補上或修正候選項目背後的專案目標。 |
| 指定任務目標 | 補上任務本身的完成目標。 |
| 要求負責人口述目標 | 要求負責人說明任務或專案目標。 |

相關版本：

- `AM-IMP-2026.0608.11`
- `docs/REPORT_INTERVENTION_ACTION_STANDARD.md`

## 4. 各時段報告定位

| 時段 | 主要用途 |
| --- | --- |
| 08:30 | 設定今日專案目標、任務目標、負責人口述目標要求；整理昨晚到早上的新訊息與新事件。 |
| 10:00 | 確認早上新增候選任務、排除雜訊、修正專案歸屬。 |
| 13:00 | 午間檢查上午進度，更新任務狀態，修正候選項目的專案或目標。 |
| 17:00 | 下班前確認未完成項目、跟催事項、待回覆事項與負責人要求。 |
| 20:30 | 完成每日結論、風險、未解任務、隔日追蹤與任務狀態收斂。 |

## 5. 10:00 / 13:00 / 17:00 追蹤報告格式

這三個時段的 follow-up confirmation report 應使用相同基本順序：

1. 今天已完成的任務。
2. 今天目前仍在進行中的任務。
3. 可選的目標確認控制，前提是該專案已有可運作的目標確認流程。
4. 候選追蹤訊息。
5. 未確認任務。
6. 寫回後的確認結果。

規則：

- 舊版不可操作的 `確認規則` 區塊不得出現在報告 UI。
- 報告中若出現候選項目，必須可以採取標準介入動作。
- 報告送出後，應留下可追蹤的寫回結果。

相關版本：

- `AM-IMP-2026.0609.04`

## 6. 08:30 報告規則

08:30 報告是每日任務管控的起點。

應處理：

- 昨晚到早上的新 LINE 訊息。
- 新事件是否應建立任務。
- 既有任務是否有狀態更新。
- 今日優先處理事項。
- 今日專案目標與任務目標。
- 需要負責人口述或補充目標的項目。

08:30 報告可以使用較完整的介入表單，讓使用者一次確認今天要追蹤什麼、不追蹤什麼。

參考原型：

```text
D:\Codex_project\AM_Core\docs\REPORT_0800_DAILY_INTERVENTION_FORM_PROTOTYPE.html
```

## 7. 20:30 報告規則

20:30 報告是每日任務狀態收斂點。

應處理：

- 今天已完成事項。
- 今天仍未完成事項。
- 需要隔日追蹤的任務。
- 今日風險與阻塞。
- 由 LINE 對話、會議、附件或報告內容判定出的狀態變更。
- 隔日優先事項。

20:30 報告不應只是摘要，應能形成任務狀態更新、隔日追蹤、風險提醒或專案進度紀錄。

## 8. 寫回與稽核

每份報告的使用者決策都應保留：

- 報告時段。
- 候選項目來源。
- 使用者選擇的 action key。
- 寫回目標，例如任務、專案、進度報告或負責人口述目標。
- 寫回結果。
- 時間戳記。
- 失敗原因與重試狀態。

## 9. 驗證要求

每次報告規則或報告頁面調整後，應檢查：

- 08:00、10:00、13:00、17:00、20:30 的定位是否清楚。
- 10:00 / 13:00 / 17:00 報告是否依標準順序呈現。
- 報告 UI 是否仍出現不可操作的 `確認規則` 區塊。
- 候選項目是否都有標準介入動作。
- action key 是否能正確寫回。
- 報告資料是否留在專案本地。
- HOZO AM 與 SevenAM 是否各自使用自己的報告資料與 Notion database。
