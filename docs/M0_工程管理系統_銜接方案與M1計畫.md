## 1. 現有 AM 架構盤點

### 1.1 生態系結構

- **AM_Core**(本 repo)是版本主控與共用程式碼中心,不是生產服務。生產專案(HOZO_AM、SevenAM)各自擁有獨立的 LINE channel、Notion workspace、Render service、GitHub repo 與 `.env`,以平行資料夾存在於 `D:\Codex_project\` 下(例:`D:\Codex_project\SevenAM\line-oa-webhook`)。
- 專案在 `config/projects.json` 註冊(projectKey、localPath、manifest 路徑、資料邊界宣告)。
- 共用改進以 `versions/AM-IMP-YYYY.MMDD.NN` 升級包發佈,分別安裝進各專案;`tools/` 提供對齊稽核。
- 新專案建立程序已有 SOP:`docs/NEW_PROJECT_DEPLOYMENT.md`。

### 1.2 執行時架構(core/runtime-template)

- **Runtime**:純 Node.js(ESM、無框架)。`src/server.js`(~1,280 行)為 webhook 主服務;`src/control-api.js`(~1,930 行)以 `--import` 掛載,提供 `/control/*` API 與 HTML 報表頁,以 Bearer token / control key 驗證。
- **Webhook 流程**:`POST /webhook/line` → 驗 `X-Line-Signature` → 逐事件非同步處理 → 取得發送者/群組 profile → 寫入 Notion。
- **儲存**:Notion 為唯一結構化儲存。以環境變數綁定多個 data source(Conversations、Messages、Attachments、Tasks、Codex Commands、Judgment Rules…)。照片/檔案由 LINE content API 下載原檔後,經 Notion File Upload API 上傳。內建資料隔離 guard(`assertHozoNotionTarget`),防止寫入非本專案的 Notion 目標。
- **部署**:各專案自有 Render service;排程工作(每日報表、每小時任務對帳)以 Render cron 執行 `scripts/` 下的 Node scripts。
- **命名慣例**:環境變數以專案前綴區分(`HOZO_*`、`SEVEN_*`);機器人稱謂如 HOZO Jr.、Seven Jr.。

## 2. 銜接方案

### 2.1 成立第三個 AM 專案:工程 AM(建議代號 BuildAM)

Seven 已決定為工程專案另建 LINE OA。這與 SPEC §6「工程模組以獨立模組/命名空間開發、不得影響 HOZO/Seven AM」完全相容,且是隔離度最高的作法:

- 依 `NEW_PROJECT_DEPLOYMENT.md` 建立第三專案,擁有自己的 LINE OA、Notion workspace、Render service、GitHub repo、`.env`。
- HOZO AM 與 Seven AM 的程式碼與資料**零接觸**,天然滿足「不得受影響」。
- 專案資料夾與 AM_Core **平行**:`D:\Codex_project\BuildAM\line-oa-webhook`(不放在 AM_Core 內,因 AM_Core 禁止存放 `.env`、tokens、生產資料)。
- 在 `config/projects.json` 註冊 `BUILD_AM`,環境變數前綴 `BUILD_*`。

### 2.2 版本策略:最小種子(待確認決策點)

工程 AM 的職責是工程管理模組,不是一般會話助理。建議:

- 以 `core/runtime-template` 為種子(webhook 收訊、Notion 寫入、control-api、部署骨架),將 `HOZO_*` 改為 `BUILD_*`。
- 只安裝基礎升級包:`AM-IMP-2026.0608.01`(資料隔離 guard)、`AM-IMP-2026.0608.05`(manifest/records),建立 manifest 保留日後補齊空間。
- **不**追齊全部 70+ 個版本包(多為會話任務抽取/報表功能,與工程模組無關,全裝會拖慢 M1 並增加雜訊)。
- 替代方案:完整依序安裝全部版本(工程 AM 同時具備一般 AM 能力)——時程會超出 M1 的 2 週。

### 2.3 結構化資料儲存:沿用 Notion(待確認決策點)

工程實體(Project、Space、WorkItem、FeedbackTicket、ChangeOrder、Message 掛載、Meeting、Task、GroupBinding)各建一個 Notion data source:

- 優點:沿用既有 notionRequest/隔離 guard/部署模式;Seven 可直接在 Notion 檢視與手動修正;無需自建資料庫的備份與維運。
- 風險與對策:
  - 流水號唯一性 → 由系統生成,單一 service 內序列化產號,寫入前查重。
  - Notion API 速率/查詢速度 → 工程訊息量(每日數十~數百則)遠低於限制;儀表板頁面做短期快取。
- 替代方案:Render Postgres 或 SQLite(查詢快,但失去 Notion 可視性、增加維運)。第一期不建議。

### 2.4 照片儲存:Google Drive 原圖(答覆 SPEC 待決定 #4)

現行做法是把 LINE 原檔上傳 Notion。工程照片需保留原圖且量大,Notion 不適合作原圖庫。建議:

- 原圖上傳 **Google Drive**(Seven 的工作流以 Google Workspace 為主),資料夾結構「專案/空間/工項/日期」,由系統在確認掛載後自動歸位。
- Notion 端存 Drive fileId + 檢視連結(+ 縮圖供佇列介面顯示)。
- 需要:一個 GCP service account 或 OAuth 憑證與指定的 Drive 共用資料夾(M3 前備妥即可;M1 先落地暫存)。

### 2.5 群組路由層(SPEC §3.3、§6)

- 新增 GroupBindings data source:LINE 群組 ID → project_id + 群組角色(設計師/工班/內部)+ 工種(工班群限定)。
- webhook 收訊後先查路由:命中 → 自動取得專案與工種兩層掛載;未綁定群組 → 訊息照收並進「待確認佇列」標記「未綁定群組」。

### 2.6 AI 初判

- 使用 Claude API,模型建議 `claude-haiku-4-5`(成本敏感場景,SPEC §6 允許)。
- Prompt 帶入該專案的空間清單與工項清單(依 F1 要求),輸出 JSON:space 推測、work_item 工種、訊息類型、FeedbackTicket 草稿建議、信心度。

### 2.7 確認佇列 Web 介面

- 沿用 control-api 模式:同一 Render service 上的 HTML 頁 + key 驗證,行動裝置可用。
- 工程夥伴為主要操作者,Seven 審核抽查;每筆記錄操作人與時間;已處理項目離開佇列;提供已確認項目審核視圖與修正記錄。

## 3. M1 開發計畫(~2 週)

### 3.0 前置(需 Seven 提供,開工前)

1. 建立工程 AM 的 LINE OA 與 Messaging API channel,提供 channel access token / channel secret。
2. 建立(或指定)工程 AM 的 Notion workspace 與 parent page,提供 integration token。
3. 建立 Render service 與 GitHub repo(或授權由 Claude Code 建立)。
4. 草悟道館(HZ)與茲心園(ZS)的空間清單(樓層/房號/公區,含 ZS 的 D 區/J 棟)。
5. 建立測試 LINE 群組並邀請機器人入群。
6. 確認本文件的決策點(§4)。

### 3.1 第 1 週:骨架 + 資料模型 + 路由

- 建立 `D:\Codex_project\BuildAM\line-oa-webhook`:以 runtime-template 種子、改 `BUILD_*` 前綴、去除 HOZO 專屬邏輯、部署至 Render 並通過 `/health`。
- 註冊 `config/projects.json`、建立 manifest、安裝基礎包(0608.01、0608.05)。
- Notion schema 建置 script:Projects、Spaces、WorkItems、GroupBindings、Messages(含掛載狀態/目標/AI 初判欄位)、FeedbackTickets(M1 僅草稿欄位,生命週期屬 M2)。
- 匯入 HZ/ZS 專案與空間種子資料。
- 群組路由層:綁定管理 + 訊息進入時自動掛專案/工種。

### 3.2 第 2 週:AI 初判 + 確認佇列

- AI 初判 pipeline:訊息落庫後呼叫 Claude API,寫回初判結果與信心度;失敗重試與低信心 fallback。
- 確認佇列 Web UI:依時間列出待確認、顯示初判、一鍵確認/修改掛載/標一般對話、高信心批次確認、操作人與時間記錄。
- 審核視圖:已確認項目列表,Seven 可修正且保留修正記錄。
- 驗收演練:測試群組發 20 則混合訊息,對照 F1 驗收標準(收集 100%、初判 ≥ 70%、3 分鐘完成確認)。

### 3.3 M1 驗收

依 SPEC F1 驗收標準,以茲心園真實群組訊息測試,Seven 驗收通過後才進入 M2。
