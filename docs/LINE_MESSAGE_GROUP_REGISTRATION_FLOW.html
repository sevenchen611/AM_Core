LINE 新群組與訊息進入 AM 系統的資料流

這份說明回答兩件事：新 LINE 群組第一次傳訊息進來時，系統目前會寫入哪些 Notion 表； 以及為了讓權責指派、通知候選、報表後續可用，還有哪些衍生表應該同步建立或刷新。

先回答問題： 不是只有 `HOZO LINE 群組選項` 這一張表會有同步缺口。 只要是從 `LINE 對話主檔` 或 `LINE 訊息紀錄` 衍生出來的輔助表，都需要有「可重跑的 upsert/reconcile 同步」。 目前最明顯的缺口是權責流程用的 `LINE 群組選項` 和 `LINE 群組成員選項`。

目前資料流總覽

1. LINE Webhook

收到事件

LINE OA 收到文字、檔案、圖片、影片或其他事件。

2. 對話主檔

找或建立對話

如果是新 group/room/user，會在 `LINE 對話主檔` 建立對話頁。

3. 訊息紀錄

建立每則訊息

每個 LINE event/message 會寫入 `LINE 訊息紀錄`，並關聯對話主檔。

4. 附件/指令

依類型分流

非文字訊息可建立附件紀錄；助理指令可建立 Codex command queue。

5. 衍生同步

後續判斷與權責

任務、進度、群組選項、群組成員、權責候選都屬於後續衍生流程。

新群組第一次出現時，應該分派到哪些表

資料表 用途 目前狀態 新群組時應該做什麼
LINE 對話主檔 一個 LINE 對話一筆，保存 Group ID / Room ID / User ID、對話名稱、監控狀態、最後訊息時間。 已自動處理 如果對話不存在，webhook 應建立新對話頁；如果存在，更新最後訊息、預覽、訊息數。
LINE 訊息紀錄 保存每一則 LINE event/message，關聯到對話主檔。 已自動處理 每則訊息都建立一筆，含訊息 ID、文字內容、來源、發話者、排序時間、原始 payload。
LINE 附件紀錄 保存圖片、影片、檔案、音訊等附件資訊，關聯到訊息與對話。 條件式自動處理 只有非文字且有附件內容時建立；文字訊息不用建立。
Codex 指令佇列 保存不能直接執行或需要 Codex 接手的 LINE 指令。 條件式自動處理 只有訊息符合助理觸發詞或 command 條件時建立；一般訊息不用建立。
LINE 群組選項 權責指派時可選的群組候選清單，讓使用者從專案縮小到群組。 目前有缺口 新 group/room 出現在對話主檔時，應 upsert 一筆群組選項，保存顯示名稱、GroupID/RoomID、專案分類、來源對話頁、訊息數、最後訊息時間。
LINE 群組成員選項 權責指派時可選的群組成員候選清單，讓使用者從群組縮小到負責人。 目前有缺口 新群組內有人發話時，應依 `群組 + UserID` upsert 成員選項，並關聯到 LINE 群組選項。
權責定義 定義每個專案或責任項目的主要群組、主要負責人、備援負責人與 LINE 發送結果。 需刷新候選 新群組不一定新增權責項目，但應在群組選項補齊後重跑候選刷新，讓符合專案分類的權責項目看得到新群組。
總控任務庫 / 專案進度報表庫 把有行動價值的訊息整理成任務、進度、風險或決策。 後續判斷 不是每個新群組都要建立任務。只有訊息內容被判斷為任務、風險、進度或決策時才建立或更新。

一次性建置

setup-responsibility-owner-narrowing.js

`seedResponsibilityOwnerNarrowingData` 會讀 `LINE 對話主檔` 中的群組/聊天室， 然後建立 `LINE 群組選項`、`LINE 群組成員選項` 和初始 `權責定義`。

關鍵流程：查對話主檔 -> normalizeConversationOption -> createPage 到群組選項表。

定期刷新但不補群組

sync-responsibility-candidates.js

這支 script 會讀現有 `LINE 群組選項` 和 `LINE 群組成員選項`， 用來更新 `權責定義` 裡的候選群組、候選負責人、候選數量與選擇狀態。

它目前不會回頭掃 `LINE 對話主檔` 補新增群組。

結論

`LINE 對話主檔` 和 `LINE 訊息紀錄` 是 ingestion 層，現在已經能自動接住新群組與新訊息。 `LINE 群組選項`、`LINE 群組成員選項`、`權責定義` 是權責治理層，目前需要新增可重跑的 reconciliation/upsert 流程。

下一個合理升級是建立一個共用規則：從對話主檔補齊群組選項，從訊息紀錄補齊群組成員選項， 再重跑權責候選刷新。這個規則應同時套用到 HOZO_AM 和 SevenAM。
