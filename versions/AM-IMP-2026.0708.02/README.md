# AM-IMP-2026.0708.02 — BuildAM 會議引擎改 AssemblyAI(講者分離＋反問與會資訊)

將 BuildAM 的會議記錄管線(F5 的一環)從「Gemini 直接轉寫」升級為
**AssemblyAI 轉寫＋講者分離**,並加入「收到錄音先反問與會者/主題」的互動,
產出**署名逐字稿**與署名摘要。開發於 2026-07-08,取代 `AM-IMP-2026.0708.01`
的會議管線部分,其餘 F1–F5 能力不變。

## 為什麼升級

舊管線(Gemini 直轉)分不出誰講的、工程術語易聽錯。Seven 導入 AssemblyAI 後,
可在轉寫前提供與會者姓名與**發言順序**、會議主題,顯著提升會議記錄品質。

## 能力範圍

- **反問狀態機**:綁定群組收到會議錄音 → 原檔即刻存 Drive 留底 → 葉小蝸反問
  兩題(①參與者:性別/角色/職務/**發言順序** ②會議主題)。群組**下一則文字**
  在進 AI 初判前被攔為「與會資訊答覆」。30 分鐘未回覆 → 逾時後備:仍以
  講者 A/B/C 直接整理,不吞錄音。
- **AssemblyAI 轉寫**:`speech_models: ["universal-3-5-pro","universal-2"]`
  (旗艦優先、缺則退回廣泛可用版)、`speaker_labels: true`(講者分離)。
  省略 `language_code` 讓中英夾雜原生切換。人名＋工程術語庫進 `keyterms_prompt`
  (≤100 詞),主題＋與會者進 `prompt`。原始 key、**不加 Bearer**。
- **署名對齊**:依 utterances 首次發言順序把 Speaker A/B/C… 對上使用者提供的
  與會者順序,產出「名字(職務):內容」逐字稿。
- **收斂**:Gemini 讀署名逐字稿 → 摘要/決議/待辦皆用真名。摘要失敗不吞逐字稿。
- **落地**:會議記錄頁(講者對照、摘要、決議、checkbox 待辦、**署名逐字稿**、
  錄音回放連結)+ 待辦任務展開 + 摘要發布回群。
- **安全後備**:未設 `ASSEMBLYAI_API_KEY` 時,自動退回舊 Gemini 直轉流程,
  行為與 0708.01 完全一致(可零風險先部署、後補 key)。

## 與 AM 家族的關係

會議引擎為可攜模組,未來工程型 AM 專案沿用同一互動與 AssemblyAI 設定即可。
刻意**不使用** AssemblyAI 內建 Speaker Identification(對應真名)——即時文件
未能驗證其 JSON schema(較新/可能 beta),改以「發言順序對齊＋姓名餵摘要」
達成一樣的署名效果,且更穩定。摘要引擎續用 Gemini(可日後換 LLM Gateway)。

## 已知限制

待補會議狀態為記憶體暫存(鍵=groupId);Render 於「等待與會答覆」空檔休眠重啟
會遺失待補狀態——但錄音原檔已存 Drive,重傳一次即可再觸發。適合目前小量會議。
